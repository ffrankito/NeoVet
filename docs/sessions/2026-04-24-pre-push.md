## Problema 1: `DropdownMenuLabel` rompía al abrir el menú de usuario

**Archivo:** [crm/src/components/admin/user-menu.tsx](../../crm/src/components/admin/user-menu.tsx)

**Error:**
```
Uncaught Error: Base UI: MenuGroupRootContext is missing.
Menu group parts must be used within <Menu.Group>.
```

**Qué pasaba:** el `DropdownMenu` del proyecto NO es de Radix (como en la mayoría de ejemplos de shadcn), sino de **Base UI** (`@base-ui/react/menu`). En Radix, `DropdownMenuLabel` funciona suelto; en Base UI, `MenuPrimitive.GroupLabel` requiere un `Menu.Group` como padre o tira ese error de contexto. Copié el patrón equivocado de memoria.

**Solución:** envolver el label en `DropdownMenuGroup`.

```tsx
<DropdownMenuContent>
  <DropdownMenuGroup>
    <DropdownMenuLabel>{email}</DropdownMenuLabel>
  </DropdownMenuGroup>
  <DropdownMenuSeparator />
  {/* ... */}
</DropdownMenuContent>
```

**Ejemplo paralelo:** es como pedir "el gerente de este sector" en un supermercado sin estar en ningún sector. Necesitás primero entrar al pasillo (Group) antes de poder pedir al gerente del pasillo (GroupLabel).

**Lección transferible:** antes de copiar un patrón de shadcn, abrir el archivo del primitivo local. shadcn es una plantilla — cada proyecto puede estar construido sobre Radix, Base UI, o algo propio, y las APIs no son idénticas.

## Problema 2: `package.json` silenciosamente downgradeado

**Archivo:** `crm/package.json` + `crm/package-lock.json` (sin commitear)

**Error:**
```
drizzle.config.ts(1,10): error TS2305:
Module '"drizzle-kit"' has no exported member 'defineConfig'.
```

**Qué pasaba:** aparecía el error en cada `tsc --noEmit` de la sesión. Al investigar antes del commit, descubrimos que `crm/package.json` tenía tres versiones distintas al último commit:

| Paquete | Commit `main` | Instalado | Movimiento |
|---|---|---|---|
| `drizzle-kit` | ^0.31.9 | 0.18.1 | -13 minors |
| `resend` | ^6.10.0 | 6.1.3 | -9 minors |
| `next` | 16.1.6 | ^16.2.4 | +1 minor (ok) |

`drizzle-kit 0.18` no exporta `defineConfig` — por eso el error. También rompía `npm run db:generate` y `npm run db:migrate`. Alguien (o alguna herramienta) corrió `npm install` con constraints distintos y npm resolvió versiones viejas, reescribiendo el lockfile.

**Solución:**
```bash
git checkout -- crm/package.json crm/package-lock.json
cd crm && npm install
git checkout -- package-lock.json    # revertimos también el lockfile reformateado
```

Después de eso: `drizzle-kit: 0.31.9`, `resend: 6.10.0`, `next: 16.1.6`. Type-check limpio.

**Ejemplo paralelo:** es como si alguien usara tu alacena y reemplazara la harina 0000 por harina leudante sin avisarte. Todo parece igual por fuera, pero la próxima vez que hornees pan te sale un bizcochuelo.

**Lección transferible:** después de un `npm install` inesperado, siempre comparar:
```bash
node -e "console.log(require('./node_modules/<pkg>/package.json').version)"
```
contra lo que dice `package.json`. Si difiere más que el patch, investigar antes de seguir. No confiar ciegamente en `npm install` — puede reescribir el lockfile de formas no triviales.

## Problema 3: script TS no podía conectarse a Supabase aunque `npm run dev` sí podía

**Archivo:** [crm/scripts/inspect-db.ts](../../crm/scripts/inspect-db.ts) (y después `seed-dashboard-smoke.ts`)

**Error:**
```
PostgresError: password authentication failed for user "tomas"
```

**Qué pasaba:** dos bugs apilados.

1. **Dotenv cargaba el archivo equivocado.** El script tenía `import "dotenv/config"` que por defecto carga `.env` (no `.env.local`). Next.js carga `.env.local` automáticamente en dev, así que la web funcionaba. El script no.

2. **`postgres.js` en tsx + Windows parseaba la URL raro.** Incluso después de apuntar a `.env.local`, el error cambió a `user "tomas"` — mi usuario de Windows. Algo en la combinación postgres.js + tsx + Windows interpretaba mal la URL (probablemente por el `.` al final del password `Fz43768194.`) y caía en el fallback de usuario del sistema.

**Solución:** cargar explícitamente `.env.local` y pasar config desestructurada en vez de URL:

```ts
import { config } from "dotenv";
config({ path: ".env.local", override: true });

import postgres from "postgres";
const url = new URL(process.env.DATABASE_URL!);
const client = postgres({
  host: url.hostname,
  port: Number(url.port),
  user: decodeURIComponent(url.username),
  password: decodeURIComponent(url.password),
  database: url.pathname.slice(1),
  ssl: "require",
  max: 1,
});
```

**Ejemplo paralelo:** Next.js y tu script son dos carteros con reglas de ruteo distintas. Next.js lee mejor letras cursivas (sabe que `.env.local` es el sobre bueno); tu script solo lee imprenta mayúscula (busca `.env`). Si no le dictás la dirección correcta, te devuelve el paquete.

**Lección transferible:** los scripts de Node puros NO comparten el loader de env de Next.js. Si un script falla donde `npm run dev` funciona, el primer sospechoso es qué `.env*` se está cargando. Y si una URL con credenciales se parsea distinto en contextos distintos, pasar los campos explícitos evita ambigüedad.
